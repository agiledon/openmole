# 坏味道驱动重构元规约（BDR Specification）

**版本**：0.2.7  
**状态**：草案 — 待维护者确认  
**依据**：[constitution.md](./constitution.md) **v1.5**、[badsmells.md](./badsmells.md) **v0.6.0**、[analysis.md](./analysis.md)（与任务三角校验时同步更新）

---

## 1. 范围与目标

### 1.1 范围

本文件是 BDR **元规约**：规定 **[`badsmells.md`](./badsmells.md)** 中 **坏味道条目** 的 **描述格式、编号规则、优先级维度与验收口径**。  

**具体坏味道清单** **必须** 写入 **`docs/bdr/badsmells.md`**（命名与位置以 [constitution.md §2](./constitution.md#2-bdr-权威工件与坏味道规约文档) 为权威）；本文件 **不重复** 承载具体条目。

### 1.2 目标

- **G1**：`badsmells.md` 中每条坏味道 **可追踪**（稳定 ID）、**可验收**（消除标准不含糊）。  
- **G2**：与 [constitution.md](./constitution.md) **§3 第一性原则** 对齐，并能在 [tasks.md](./tasks.md) 中 **一对一或多对一** 映射为任务。  
- **G3**：与 SDD 规约 **无冲突**；若某条「坏味道」的消除会改变 **对外 API 或行为**，该条必须 **同时** 标记为 **SDD 联动项**（见 **§5**）。

## 2. 术语

| 术语 | 定义 |
|------|------|
| **坏味道** | 违背宪法 **§3** 中清晰、一致、可读、复用、扩展、健壮、安全、简单之一项或多项的代码结构问题；通常可对应 Fowler《重构》中的常见气味标签。 |
| **`badsmells.md`** | **`docs/bdr/badsmells.md`**：坏味道环节 **唯一** 正式规约文档，承载 **具体条目** 与元数据；版式与其它 BDR 文档一致。 |
| **规约条目** | 本文件 **§4** 规定的、写入 **`badsmells.md`** 的 **单条坏味道** 最小结构。 |
| **BDR 任务** | [tasks.md](./tasks.md) 中的可交付单元，须引用 **`badsmells.md`** 中至少一条的 ID 或章节。 |
| **`BS-CONSIST-*` ID** | 坏味道 **稳定 ID 前缀**，映射宪法 **§3.2 一致**（类型系统与实现、导出面、错误处理风格等与团队约定对齐）。与 **`BS-CLARITY-*`**、**`BS-SIMPLE-*`** 等并列，**非** 单独文档；条目仍写在 **`badsmells.md`**。 |

## 3. 优先级维度（与宪法 §3 顺序一致）

`badsmells.md` 中的大类排序 **建议** 与宪法 **第一性原则** 顺序一致：

1. **清晰**（结构、职责、边界、耦合）  
2. **一致**（设计/实现、风格、错误处理、接口与通信、对外访问）  
3. **可读**（命名、注释）  
4. **复用**（DRY、专家、迪米特、SRP、ISP）  
5. **扩展**（OCP、DIP、接口与版本、可测性）  
6. **健壮**（分级错误与异常、日志、避免非预期崩溃）  
7. **安全**（输入/机密/权限/依赖等常见风险）  
8. **简单**（奥卡姆剃刀、避免过度设计）

**说明**：「优先级」用于 **排序与裁剪范围**，不表示可跳过测试与标准步骤。

## 4. 清单条目格式（写入 `badsmells.md` 的规约要求）

每条坏味道 **至少** 包含：

1. **ID**：稳定标识，建议 `BS-类别-序号`（如 `BS-CLARITY-001`、`BS-CONSIST-001`）或连续编号并在 `analysis.md` 中维护 **ID 映射表**。  
2. **标题**：短句概括。  
3. **位置**：主要涉及的 **模块 / 文件 / 符号**（可多条）。  
4. **描述**：现象与为何构成坏味道（可引用 Fowler 标签）。  
5. **对齐原则**：指向宪法 **§3** 中的子项（清晰 / 一致 / 可读 / 复用 / 扩展 / 健壮 / 安全 / 简单 等）。  
6. **消除标准（验收）**：如何判断「已消除」。  
7. **风险与约束**：是否涉及 **行为变更** 等；若 **是行为变更** → 标记 **SDD 联动**（见 **§5**）。

可选：**建议重构手法** — **非强制**，以任务设计为准。

---

## 5. SDD 联动项（强制标注）

若消除某坏味道 **可能或必然** 改变公开 **API** 或 **可观察行为**，则该条目在 **`badsmells.md`** 中必须 **`[SDD]`** 或单列 **「须先修订 SDD」**；对应 [tasks.md](./tasks.md) 任务 **首条子项** 为完成 SDD 修订与确认。

---

## 6. 与 `badsmells.md` 的唯一性（其它叙述材料）

**具体坏味道清单** **仅** 写入 **`docs/bdr/badsmells.md`**（见 [constitution.md](./constitution.md) **§2**）。仓库内其它 markdown 若存在叙述性内容，**不得**替代 **`badsmells.md`** 作为 BDR 门禁与任务追溯载体。条目迁入或重写时须 **升版 `badsmells.md`** 并走 [analysis.md](./analysis.md) **更新 tasks.md**。

---

## 7. 修订历史表与「提交版本」（门禁）

下列三份文档文末 **修订历史** 表 **必须** 包含列 **提交版本**（列顺序：**版本**、**日期**、**提交版本**、**摘要**）：

- **[`badsmells.md`](./badsmells.md)** — **识别坏味道** 环节；  
- **[`tasks.md`](./tasks.md)** — **分解任务** 环节；  
- **[`analysis.md`](./analysis.md)** — **分析** 环节。

**填写规则**：

1. **取值**：在该次文档修订 **与之一同进入仓库的 Git 提交** 落稿时，执行 **`git rev-parse HEAD`**，将输出的 **40 字符完整 commit SHA** 写入 **本条修订历史行** 的 **提交版本** 列（可用反引号包裹，如 `` `abc…` ``）。  
2. **时机**：凡 **升版** 文档页眉版本号，或 **在修订历史中新增一行**，**必须** 同步填写 **提交版本**；三份文档 **不要求** 共用同一 SHA（允许分次提交），但 **每一行** 须对应该次提交的真实值。  
3. **历史回填**：本门禁引入 **之前** 的修订历史行，**提交版本** 填 **`—`**（未记录）。  
4. **仅本地未提交**：暂填 **`—`**，在 **下一次** 与文档一并提交前改为实际 SHA。

---

## 8. 验收标准（本元规约自身）

1. **`badsmells.md`** 中每条 **遵循 §4**。  
2. [tasks.md](./tasks.md) 每个重构任务 **可追溯到 `badsmells.md`**。  
3. **`badsmells.md` 变更后** 已通过 **analysis** 同步任务文档。  
4. [analysis.md](./analysis.md) 中 **无未解决的** 宪法 / 元规约 / `badsmells.md` / 任务 **冲突**。  
5. **`badsmells.md`**、**`tasks.md`**、**`analysis.md`** 的修订历史 **含 §7 所列「提交版本」列**，且新增行 **遵守 §7 填写规则**。
